# TravelPlan AI — V1 Roadmap

**V1 goal:** Ship a trustworthy MVP that reliably turns any reasonable trip brief into a
complete, well-structured day-by-day itinerary — with graceful error handling, a polished
core UX, and a live production URL on Vercel.

---

## Definition of "Done" (V1 acceptance criteria)

- [ ] Any valid trip brief (destination + any subset of optional fields) produces all 8 plan
      sections — title, overview, weather, transport, daily schedule, food, budget table, tips.
- [ ] The model never silently skips a section, invents prices without basis, or ignores the
      stated pace/budget level.
- [ ] Every failure mode (missing key, network error, vague brief, timeout) shows a helpful
      message — no blank screen, no uncaught JS error.
- [ ] The app is usable on a 375 px mobile viewport without horizontal scroll or cut-off text.
- [ ] `tsc --noEmit`, `npx eslint src/`, and `npm run build` all exit clean.
- [ ] The production Vercel URL serves a complete plan within 120 s for a standard brief.

---

## ⭐ Phase 1 — Output Quality & Prompt Stability

> **Start here. Everything downstream depends on the core output being good.**

**Goal:** The generated plan is consistently complete and specific across diverse trip types.

### Tasks

- [ ] Run 8–10 diverse test briefs end-to-end and note any regressions:
  - Short domestic weekend (no dates, no origin)
  - 2-week international multi-city (all fields filled)
  - Solo budget backpacker
  - Family with young kids
  - Vague destination ("somewhere warm in Southeast Asia")
  - Luxury honeymoon
  - Trip with restrictive notes (vegan, no flights, mobility issues)
  - Very short trip (1 day / city break)
- [ ] For each test: verify all 8 sections appear and contain specific place names (not
      "a local restaurant").
- [ ] If sections go missing or are generic, tighten the system prompt — add explicit
      "if X is missing, assume Y and say so" guards, not new features.
- [ ] Verify the budget table always appears and the TOTAL row is present.
- [ ] Confirm `stopWhen: stepCountIs(12)` is sufficient for a 2-week multi-city trip; bump
      if the model cuts the plan short on complex briefs.
- [ ] Verify the model states its assumptions once at the top when dates or origin are absent,
      then proceeds — not mid-plan hedging.

### Verification

Run 5 briefs covering different lengths and styles. All 5 must produce all 8 sections with
specific named venues and a populated budget table. Zero generic fallback phrases like
"a popular attraction" or "prices vary."

## Phase 2 — Error & Edge-Case Handling

**Goal:** The app tells the user what happened and what to do for every failure mode.

### Tasks

- [ ] Test with `ANTHROPIC_API_KEY` unset on the server — verify the 500 response is
      caught by the `error` object in `useChat` and the red error div renders with a
      clear, human-readable message (not a raw JSON string).
- [ ] Improve the error message fallback in `page.tsx` — distinguish "key missing" from
      "network error" if possible; at minimum ensure the message is a full sentence.
- [ ] Test the 120 s timeout path (`maxDuration`): submit a brief, wait — does the UI
      eventually show an error or just hang? Add a timeout-specific message if it hangs
      silently.
- [ ] Test stop → clear → resubmit: verify `setMessages([])` + `sendMessage()` starts a
      clean request with no leftover state.
- [ ] Test mid-stream connection drop (kill the dev server tab): verify partial plan text
      stays visible and the spinner stops.
- [ ] Verify the Submit button stays disabled while `busy` so double-submit is impossible.
- [ ] Test whitespace-only destination — should be blocked at the form level (already
      guarded; just confirm).

### Verification

Trigger each scenario above. None produce a blank main panel or an unhandled React error.
Each shows a message the user can act on.

## Phase 3 — UX Polish

**Goal:** The app feels smooth and trustworthy on desktop and mobile.

### Tasks

- [ ] Test on a 375 px viewport (iPhone SE): check sidebar↔main stacking, form tap
      targets (≥ 44 px), and plan markdown readability.
- [ ] Verify markdown renders correctly *during* streaming — headings, tables, and lists
      should appear as they arrive, not as raw `#` or `|` characters.
- [ ] On mobile, auto-scroll to the plan panel when streaming begins so the user doesn't
      stare at the form while waiting.
- [ ] Review the tool activity chips: confirm they appear for every tool call and turn
      green (✓) when the tool returns — no chips stuck in spinner state after the plan
      is complete.
- [ ] Check prose max-width in the plan panel — long markdown tables on a very wide
      screen should not overflow horizontally.
- [ ] Verify color contrast on all small text (slate-400, slate-500 labels) meets WCAG AA.
- [ ] Polish the empty state copy if needed — the current "Fill in your trip brief…"
      paragraph is fine; just confirm it reads well after seeing real output.

### Verification

Open on a 375 px emulated viewport in Chrome DevTools. Submit a trip brief. The form,
activity chips, and streaming plan all render without horizontal scroll or clipped text.
Tool chips all resolve to ✓ after the plan completes.

## Phase 4 — Real-Data Tools

**Goal:** Google Places and Amadeus are wired and tested; degradation to web search works
cleanly when keys are absent.

### Tasks

- [ ] Obtain a Google Places API (New) key; enable the Places API (New) in Google Cloud
      console; add `GOOGLE_MAPS_API_KEY` to `.env.local`.
- [ ] Submit a test brief and confirm `searchPlaces` returns real results with ratings,
      hours, and coordinates in the tool chip output.
- [ ] Check that the model cites specific venue names and ratings from Places data in the
      itinerary.
- [ ] Obtain Amadeus self-service test-tier credentials; add `AMADEUS_CLIENT_ID` and
      `AMADEUS_CLIENT_SECRET` to `.env.local`.
- [ ] Submit a brief with a clear origin city and verify the two-step hotel flow
      (city → IDs → prices) completes without error.
- [ ] Verify flight price ranges appear in the "Getting there" section when both origin
      and dates are provided.
- [ ] Test each optional key individually removed — confirm the tool returns
      `{ configured: false }` and the model substitutes a web search without crashing or
      producing a visible error in the plan.
- [ ] Check `.env.example` has all five optional vars documented with placeholder values
      and source links.

### Verification

With all three optional keys set, a "New York → Tokyo, 7 days, mid-range" brief produces:
real venue names with ratings in the daily schedule, a flight price range in the transport
section, and a hotel option in the budget table. Remove each key one at a time; the plan
still completes — just without that data source.

## Phase 5 — QA Gates

**Goal:** The codebase is type-clean and the production build is green before deploy.

### Tasks

- [ ] Run `tsc --noEmit` and fix every error (pay attention to `any` casts and implicit
      return types on tool `execute` functions).
- [ ] Run `npx eslint src/` and fix all warnings — unused imports, missing keys in JSX
      maps, etc.
- [ ] Run `npm run build` (Turbopack production build) and resolve any warnings or errors
      in the output.
- [ ] Address any Next.js 16 deprecation notices printed during build.
- [ ] After build, run `npm start` locally and do one full golden-path test (fill form →
      stream plan → verify all 8 sections) against the production bundle.

### Verification

All three commands (`tsc --noEmit`, `npx eslint src/`, `npm run build`) exit with code 0
and zero warnings treated as errors. The production bundle golden-path test passes.

## Phase 6 — Vercel Deploy

**Goal:** The app is live on a production Vercel URL with all env vars set.

### Tasks

- [ ] Run `vercel link` to connect the local repo to a Vercel project.
- [ ] Add `ANTHROPIC_API_KEY` to Vercel → Project Settings → Environment Variables
      (Production scope).
- [ ] Add optional keys (`GOOGLE_MAPS_API_KEY`, `AMADEUS_*`) to Vercel if they are ready.
- [ ] Run `vercel deploy --prod` and confirm the build log is clean.
- [ ] Open the production URL in a browser and submit a test brief — verify streaming
      works (no buffering delay before text appears).
- [ ] Confirm the `maxDuration = 120` setting is respected on Vercel (it is set on the
      route; no additional config needed for Vercel's default 300 s function timeout).
- [ ] Test the error state on production: temporarily remove the API key from Vercel env,
      redeploy, verify the error message renders — then restore the key.
- [ ] Ensure `.env.local` and all real API keys are listed in `.gitignore` and are NOT
      in the deployed source.

### Verification

The production URL produces a complete plan for a standard brief. The streaming text
appears within a few seconds of submit (not all at once at the end). The error message
displays correctly when the key is absent.

## Parked for V2

- **Saved trips** — user accounts, named itinerary history, re-open and edit past plans.
- **Trip sharing** — public read-only link for a generated plan.
- **Interactive map** — pins for each day's stops plotted on an embedded map.
- **Follow-up chat** — ask the AI to tweak the plan ("add a day in Kyoto", "swap the hotel").
- **PDF / email export** — download or share the itinerary as a formatted document.
- **Multi-destination / road trip** — structured builder for A → B → C routes.
- **Embedded booking links** — deep links to flight/hotel partners in the plan output.
- **Mobile native app** — iOS/Android wrapper around the web app.
- **Payments / premium tier** — rate limiting, usage quotas, subscription gating.
- **Collaborative planning** — shared workspace where multiple users edit a trip together.
